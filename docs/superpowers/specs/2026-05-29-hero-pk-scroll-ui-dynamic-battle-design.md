# Hero PK 卷轴动态战斗 UI 设计

## 1. 背景与目标

当前 hero-pk 前端已经具备对战配置、回合结果展示和基础动态播放能力，但战斗演出张力仍不足。

本次目标：

1. 从“结果面板”升级为“战斗舞台”。
2. 回合事件具备实时叙事感，不再机械滚日志。
3. 提供可控节奏（暂停、单步、重放、倍速）。
4. 保持现有后端接口不变，仅做前端演出增强。

## 2. 用户场景

1. 玩家在选择英雄后发起模拟，希望看到过程而不是最终结算。
2. 玩家希望判断胜负关键节点（暴击、格挡、中毒、爆发窗口）发生在何时。
3. 玩家希望重复观看同一场战斗并控制播放速度。

## 3. 视觉方向（江湖卷轴风）

1. 视觉语义：纸张底纹、墨色正文、印章红强调色、战报式排版。
2. 字体语义：标题书卷感、正文可读性优先。
3. 叙事语义：每回合“书写”进入卷轴，战况播报同步更新。

## 4. 交互设计分节

### 4.1 战斗舞台

1. 左右角色位：展示对战双方头像。
2. 中央对阵标记：强化“PK 舞台”识别。
3. 技能特效浮层：显示斩击、破势、格挡、毒雾、回春。

### 4.2 动态播放控制

1. 播放/暂停。
2. 单步一回合。
3. 重放本场。
4. 慢/中/快三档速度。

### 4.3 实时状态反馈

1. 顶部战况播报条：输出当前回合叙事。
2. 受击血条脉冲：哪侧受击哪侧闪动。
3. 回合日志色调：危险、毒性、系统、中性分类高亮。

## 5. 技术实现

1. 数据层：使用已有 battle result logs，不改接口。
2. 状态层：新增播放状态、舞台动作状态、特效状态。
3. 渲染层：
- HTML：战斗舞台结构 + 控制区 + 浮层。
- CSS：关键帧动画 + 主题变量 + 日志色调。
- JS：回合推进时解析事件并触发对应演出。

## 6. 验收标准

1. 发起战斗后，日志逐条进入而不是一次性渲染。
2. 普攻/技能回合可以看到人物动作与特效浮层。
3. 暴击/重击/中毒/格挡在日志与播报上可识别。
4. 播放控制在任意回合均可用，重放结果一致。
5. 页面在桌面和移动端可正常显示。

## 7. 风险与约束

1. 文本关键词驱动特效，依赖日志文案稳定。
2. 若低性能设备卡顿，后续需提供“简化演出模式”。
3. SVG 头像为静态资源，后续可替换为英雄专属立绘。

## 8. 非目标

1. 不改 BattleService 核心战斗逻辑。
2. 不新增后端动画接口。
3. 不引入大型动画库。

## 9. Spec Self-Review

1. 完整性：覆盖目标、交互、实现、验收、风险。
2. 可执行性：全部基于现有静态资源与日志字段可落地。
3. 可验证性：每项都有前端可观察结果。
4. 风险可控性：明确了关键词依赖和降级策略。
