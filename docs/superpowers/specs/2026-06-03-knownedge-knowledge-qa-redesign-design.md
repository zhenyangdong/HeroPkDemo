# knownedge 主题化知识库问答重设计（方案 A 增量改造）

## 1. 背景与目标

本设计面向 knownedge 现有项目（明确排除 hero-pk 与 electronppt），在保留当前上传、解析、检索与问答骨架的前提下，增量升级为“主题化知识库问答系统”。

核心目标：

1. 支持主流文档输入：txt、md、docx、pdf、xlsx、csv。
2. 以日常问答检索为主场景，默认输出证据与出处。
3. 回答策略采用“文档优先 + 可补充推断（显式标注）”。
4. 支持主题化知识域：上传归属主题、提问按主题检索与回答。

## 2. 范围与非范围

### 2.1 范围

1. 后端：输入标准化、分段、索引、问答编排、主题过滤。
2. 前端：主题选择、问答结果证据展示、来源跳转入口。
3. 数据：主题、文档、段落、证据结果结构化。

### 2.2 非范围

1. 不改造 hero-pk。
2. 不改造 electronppt。
3. 本期不做多主题归属（单文档单主题）。
4. 本期不引入复杂分布式架构。

## 3. 总体流程蓝图

1. 输入层（Ingest）
- 接收上传或目录扫描文件。
- 统一识别文件类型和来源路径。

2. 规范化层（Normalize）
- 文本清洗、去噪、换行归一。
- 切分为可检索段落（Passage），记录来源坐标。
- 提取关键词、标签、分类辅助信息。

3. 索引层（Index）
- 维护关键词索引与段落证据索引。
- 支持增量更新与全量刷新。

4. 提问编排层（QA Orchestrator）
- 用户选择主题后发起问题。
- 先召回证据段落，再合成答案。
- 输出统一结构：结论、依据、来源、补充推断。

5. 展示层（UI）
- 提问前选择主题。
- 回答区默认显示证据卡片与来源列表。
- 支持来源定位跳转入口。

## 4. 模块边界设计

1. ingest 模块
- 输入接入、文件检测、上传落盘。

2. normalize 模块
- 解析器调度、文本规范化、切段、关键词提取。

3. index 模块
- 段落索引、检索排序、增量更新。

4. qa 模块
- Query 理解、证据召回、冲突检测、答案合成。

5. api 模块
- 对外路由与 DTO，保持控制器轻量。

## 5. 数据结构定义（建议）

### 5.1 Topic

1. topicId
2. name
3. description
4. isActive
5. createdAt

### 5.2 Document

1. docId
2. topicId
3. title
4. sourcePath
5. fileType
6. tags
7. updatedAt

### 5.3 Passage

1. passageId
2. docId
3. topicId
4. sectionTitle
5. text
6. position（段号/页号/行号）
7. keywords

### 5.4 EvidenceHit

1. passageId
2. score
3. snippet
4. reason

### 5.5 QAResult

1. answer
2. confidence（high/medium/low/unknown）
3. evidence[]
4. sources[]
5. supplementalInference[]
6. insufficient（bool）
7. topicId

## 6. 回答策略与质量规则

1. 文档优先：先证据后结论。
2. 允许补充推断：必须显式放在“补充推断”区。
3. 证据不足：拒绝硬答，并提示补充资料方向。
4. 冲突证据：输出冲突点与双侧依据，不给高置信单结论。
5. 输出模板固定：
- 结论
- 依据
- 来源
- 补充推断（可选）
- 不确定点（可选）

## 7. 主题化能力设计（新增重点）

1. 主题管理
- 支持创建主题：工作经验、美食、交通、武侠等。
- 主题可启停、可描述。

2. 上传归属主题
- 上传时必须选择主题。
- 本期规则：单文档单主题。

3. 提问按主题过滤
- 提问前选择主题。
- 检索与问答仅在该主题文档内执行。

4. 结果展示主题上下文
- 每条证据显示主题与来源。
- 错主题导致证据不足时，提示切换主题。

5. 模式扩展预留
- 本期默认单主题问答。
- 预留全库问答与跨主题对比问答入口。

## 8. 分阶段实施路线图（MVP）

### Phase 0（0.5 天）

1. 固定范围与输出协议。
2. 定义主题与文档基础结构。

验收：
1. 有明确范围声明。
2. 输出 schema 明确。

### Phase 1（2 天）

1. 统一解析输出为 Document + Passage。
2. 完成切段与来源坐标。

验收：
1. 任意支持格式可得到标准段落。
2. 每段可追溯来源。

### Phase 2（2-3 天）

1. 建立主题内检索与证据召回。
2. 支持增量 refresh。

验收：
1. Top-K 召回结果可解释。
2. 证据返回含 score 与 snippet。

### Phase 3（2 天）

1. 实现 QA 编排与冲突处理。
2. 输出统一模板。

验收：
1. 默认带证据与来源。
2. 证据不足时拒答合理。

### Phase 4（1-2 天）

1. 前端增加主题选择。
2. 增加证据卡片与来源跳转。

验收：
1. 用户可按主题提问。
2. 回答可追溯来源。

### Phase 5（1 天）

1. 构建基准问题集并做回归。
2. 跑命中率与拒答质量检查。

验收目标：
1. Top-3 证据命中率 >= 80%。
2. 高置信错误率可控。
3. 无依据硬答比例接近 0。

## 9. 风险与对策

1. 文件解析质量不稳定
- 对策：按格式做降级策略与错误回显。

2. 主题误选导致回答偏差
- 对策：证据不足时提示切换主题并列推荐主题。

3. 数据规模增长后检索性能下降
- 对策：保留索引实现替换能力，后续可平滑升级。

## 10. 设计结论

建议采用方案 A 增量改造，先把“主题化 + 证据化问答”做成稳定 MVP，再视效果按模块边界向更强检索架构演进。
